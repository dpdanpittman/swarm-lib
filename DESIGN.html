swarm-lib — Design Spec v0.1

Status: Draft — pending review · Inspired by the Inkcloud Architecture Post-Mortem ("Unix Swarm Blueprint")

A small Python + Bash library that gives agentic workflows three primitives:
  1. Atomic-rename task queueing — multiple workers can race for tasks; exactly one wins, no external broker
  2. status.json checkpointing — durable handoff so any agent can resume from a file, never from chat history
  3. A generic worker_loop.sh — polls a queue, claims a task, invokes a handler, moves the result
v0.1 substrate only. Triage (HMD), orphan cleanup, multi-host federation, and a Kanban UI deferred to v0.2+. Goal: prove the substrate by porting Tribunal's 7-skill chain onto it.

Why this exists

Problem Manifestation swarm-lib answer
Context starvation Long conversation accumulates context, hits compaction, in-flight work dies Tasks become claimable by fresh contexts; no single conversation owns the whole chain
Sync tool-call blocking Planner holds an expensive context window open while subprocess churns for 30 min The "Yield Rule": planner decomposes → writes payload to queue → exits. Consumers claim asynchronously.
Chat-history-as-state If the conversation dies, the work dies status.json is the canonical state. Chat is volatile; the file is the contract.

UNIX shops have had this discipline for 30 years (Maildir, cron + lock files, /var/spool/). swarm-lib applies it to LLM-driven agent work.

Architectural principles

1. The file system IS the orchestrator

No daemon, no broker, no external database. Directory layout + POSIX atomic rename + JSON files do all coordination. Any agent with read/write access to the queue directory is a participant.

2. The Yield Rule

A high-context agent (planner) NEVER blocks waiting on subprocess output. It decomposes → enqueues atomically → exits immediately, freeing its context. Fresh consumers claim tasks in their own short-lived contexts. The planner can be re-invoked later from status.json.

3. Atomic-rename queueing (Maildir physics)

Producers stage tasks under pending/. Consumers claim by os.replace(pending/<id>.json, claimed/<worker_id>/<id>.json). POSIX guarantees the rename is atomic on the same filesystem — two consumers racing for the same task, exactly one wins.

4. Agents are interchangeable

A worker is anything that can read a JSON task file, do work, and write a JSON result file:

  • Claude Code (Mabus) with a specific skill
  • Ollama-backed local agents (via ollama-assistant MCP)
  • n8n flows
  • Plain shell scripts
  • Future agents that don't exist yet

Task ownership lives in the filesystem path (the claimed/<worker_id>/ directory), not in a database row tied to a specific agent identity.

5. Cold-start recoverability

Any new agent picks up an in-flight workflow by reading two files:

  • portrait.md — identity continuity (from session-essence)
  • status.json — task-state continuity (from swarm-lib)

Compactions, crashes, rate limits, multi-day pauses — all become indistinguishable from a clean restart.

Directory layout

<consumer-repo>/.swarm/<run-id>/
├── BACKLOG.md           # original human intent (optional, free-form)
├── status.json          # current checkpoint state (durable handoff)
├── pending/             # tasks waiting to be claimed
│   └── <task_id>.json
├── claimed/             # tasks currently being worked on, namespaced by worker
│   └── <worker_id>/
│       └── <task_id>.json
├── done/                # successfully completed tasks
│   └── <task_id>.json
├── failed/              # quarantined tasks (worker death without clean handoff)
│   └── <task_id>.json
└── artifacts/           # task outputs — markdown, JSON, logs, anything
    └── <task_id>.<ext>

Cross-project federation: when a task spans multiple consumer repos (Mabus → n8n → Mabus), the run directory lives under ~/.swarm/<run-id>/ instead. v0.1 doesn't enforce this distinction — worker_loop.sh takes a --run-dir arg.

status.json schema

{
  "schema_version": "0.1",
  "run_id": "tribunal-r42x",
  "checkpoint": {
    "summary": "Completed plan; ready for implement stage",
    "next_step": "Invoke implement skill with plan output as input",
    "next_task_id": "t.r42x.implement",
    "resume_command": null,
    "risk": "Plan references xion-stytch-proxy hostnames; ensure consumer has env access",
    "completed_tasks": ["t.r42x.intent", "t.r42x.plan"],
    "current_worker": null,
    "timestamp": "2026-05-20T18:45:11Z"
  },
  "metadata": {
    "created_by": "mabus",
    "consumer": "tribunal",
    "tier_hint": "opus"
  }
}
Field Type Required Notes
schema_version string yes semver; lib refuses to read unknown majors
run_id string yes unique per workflow run
checkpoint.summary string yes human-readable "what was just finished"
checkpoint.next_step string yes concrete "what to do next" — agent-readable
checkpoint.next_task_id string | null yes the task_id to claim next; null = workflow complete
checkpoint.resume_command string | null no optional; must be validated against allowlist before exec
checkpoint.risk string yes known blockers or fragile state
checkpoint.completed_tasks string[] yes task_ids that have run
checkpoint.current_worker string | null yes worker_id currently holding a claim
checkpoint.timestamp ISO-8601 yes last update
metadata.* object no freeform; consumer-specific

Evolution policy: minor bumps additive only. Major bumps breaking; lib supports reading the previous major for one cycle to allow migration.

Pushback note: I removed the resume_command "blindly execute" framing from the slideshow. If consumers want a shell-resumable workflow, the command must be validated against an allowlist before exec. Default is null; LLM-driven workflows ignore it.

Task schema

{
  "task_id": "t.r42x.implement",
  "task_type": "implement",
  "run_id": "tribunal-r42x",
  "created_at": "2026-05-20T18:42:11Z",
  "created_by": "mabus",
  "depends_on": ["t.r42x.plan"],
  "payload": {
    "skill": "tribunal-implement",
    "input_artifacts": ["artifacts/t.r42x.plan.md"],
    "output_artifact": "artifacts/t.r42x.implement.md"
  },
  "tier_hint": "sonnet",
  "deadline": null
}
  • payload is consumer-specific; lib doesn't validate its shape
  • tier_hint is an optional cost-routing signal (used in v0.2)
  • depends_on is honored by worker_loop.sh — won't claim until deps are in completed_tasks
  • deadline is informational in v0.1; v0.2 may route to a scheduler worker

Claim protocol

Producer (enqueue)

from swarm_lib import claims

claims.enqueue(
    run_dir="~/src/tribunal/run/r42x",
    task_id="t.r42x.implement",
    task_type="implement",
    payload={...},
    depends_on=["t.r42x.plan"],
)

Internally: write JSON to temp file → fsync → os.replace(tmp, pending/<id>.json) → atomic publish.

Consumer (claim)

task = claims.try_claim(
    run_dir="~/src/tribunal/run/r42x",
    worker_id="mabus-tribunal-1",
    task_type_filter=["implement", "verify"],
)
if task is None:
    return  # no claimable task

do_work(task)
claims.complete(task, success=True, artifact_path="...")

Internally try_claim scans pending/, attempts os.replace per candidate, returns the first successful claim. OSError means another worker won — try next candidate.

Orphan recovery (v0.2)

If a worker dies mid-claim, the task is stuck in claimed/<worker_id>/. v0.2 adds cron-driven cleanup. v0.1 ignores this; manual cleanup is fine for now.

worker_loop.sh contract

worker_loop.sh \
  --run-dir ~/src/tribunal/run/r42x \
  --worker-id mabus-tribunal-1 \
  --handler tribunal_handle.sh \
  --task-type-filter implement,verify \
  --poll-interval 5 \
  --max-iterations 100

Loop body: try_claim → if claimed, invoke handler with task JSON on stdin → on handler exit, mark done/failed → repeat.

A handler is any executable that reads task JSON from stdin, writes its output to $SWARM_ARTIFACT_PATH, optionally enqueues follow-up tasks, updates status.json, and exits 0/non-zero.

Tribunal port plan

Today's shape

/tribunal <prompt> runs intent → plan → implement → review → verify → classify → incentive sequentially in one Claude Code context. Long probes blow the window. Compaction loses the chain.

Ported shape

~/src/tribunal/run/<run-id>/
├── BACKLOG.md           # user's /tribunal prompt verbatim
├── status.json
├── pending/ claimed/ done/ failed/
└── artifacts/
    ├── 01-intent.md
    ├── 02-plan.md
    ├── 03-implement.md
    ├── 04-review.md
    ├── 05-verify.md
    ├── 06-classify.md
    └── 07-incentive.md

Flow

  1. /tribunal <prompt> writes BACKLOG.md + enqueues intent task + writes initial status.json → exits
  2. Worker loop polls pending/ → claims intent → invokes claude -p --skill tribunal-intent
  3. Intent handler writes artifacts/01-intent.md, enqueues plan, updates status, exits
  4. Worker loop continues through plan → implement → review → verify → classify → incentive, each in a fresh context
  5. Final task completes; status.json::next_task_id = null signals workflow complete

Model tier mapping

Skill Tier Why
intent haiku Cheap, just structures the user's input
plan opus Hardest reasoning step
implement sonnet Mechanical from plan
review opus Adversarial probe quality matters here
verify sonnet Pattern matching
classify haiku Just categorization
incentive sonnet Generative but bounded

Compaction survival

At any point, if the worker dies, the next agent reads status.json, sees next_task_id, claims it from pending/, resumes. No conversation history needed.

Open questions for review

  1. Worker ID convention: PID-based (auto, transient) vs named (mabus-tribunal-1, durable). I lean named.
  2. Run directory location: per-consumer-repo <repo>/.swarm/ vs shared ~/.swarm/<run-id>/. Lean per-consumer-repo for v0.1.
  3. Task ID format: short prefix + run-id + type (t.r42x.implement) vs UUID vs ISO timestamp. Lean short-prefix-with-run-id.
  4. Atomic write pattern: temp-then-rename (safer) vs direct write (faster). Lean temp-then-rename.
  5. Schema versioning: strict refuse-unknown vs warn-and-attempt. Lean strict.
  6. License: confirmed MIT.
  7. Single source of truth for completion: status.json::completed_tasks (cached) + done/ directory (artifact-of-record). They should match; drift is a bug.

Risks

  • Partial writes: mitigated by temp+rename pattern, lib-enforced
  • Filesystem mismatch: lib validates source + dest on same mount at startup
  • Race between scan and claim: expected; OSError = lost race, loop continues
  • Compaction during status write: low risk via temp-rename atomicity
  • Schema drift: metadata.* preserved on read-modify-write

Inspiration / prior art

  • Inkcloud Architecture Post-Mortem — direct inspiration for Yield Rule + filesystem-as-orchestrator
  • Maildir (1995) — atomic rename for email delivery; same physics
  • cron + lock files + /var/spool/ — UNIX queue patterns since the 90s
  • GitHub Actions reusable workflows + workflow_call — same yield rule applied to CI/CD (per the DO-302 work)
